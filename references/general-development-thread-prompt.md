# 通用开发线程模板提示词

## 适用场景

在文档体系已经搭好之后，用户希望另起一个开发线程、接手线程或继续实现线程时，直接交付这份提示词。

## 交付方式

把下面整段提示词原样或按项目实际轻微替换占位符后交给用户。

```text
你现在进入 `<项目名>` 的新开发 / 交接线程。

你的任务不是重新发明文档体系，而是沿着现有代码和正式文档继续推进 `<本次目标>`。

请严格按下面方式工作：

一、先校准，不要直接开写
1. 先定位项目根目录：`<仓库根目录>`
2. 找到唯一入口文档：`<总入口文档路径>`；如果没有这个名字，就找等价入口
3. 按入口规定的阅读顺序，至少读取：
   - 现状文档
   - 阶段判断文档
   - 未来路径文档
   - 当前版本实施文档
   - 任务链索引
   - 本次任务涉及模块的实现事实文档
   - 过程日志最近一段
4. 读取后，用真实代码和 `git status` 校准文档，不要只信文档

二、先用几句话复述你对项目的理解
请先明确说清：
1. 现在真实已经做到什么
2. 这次目标属于修现状、补当前版本，还是推进未来链路
3. 本次会影响哪些模块
4. 哪些内容绝不能误写成已实现

三、执行规则
1. 优先直接做代码、文档和必要校验，不要长篇空谈
2. 正式文档只写稳定结论
3. 命令、测试、临时判断、备份、检查点统一写进 `<过程日志路径>`
4. 只更新与当前任务直接相关的文档，不要顺手重写整套体系
5. 如果项目已经有自己的文档命名，保留原名，只做角色映射
6. 如果发现文档与代码冲突，以真实代码和 git 状态为准，再回写文档
7. 在“功能接通后 / 校验通过后 / 准备切换方向前”这些关键节点，如果当前项目要求备份与检查点，请按项目习惯执行

四、输出要求
1. 先给出你已读取并校准的文档与模块范围
2. 再开始实现
3. 收尾时明确说明：
   - 改了哪些代码
   - 改了哪些正式文档
   - 做了哪些校验
   - 还剩什么风险或待办

五、严格边界
1. 不要把未来计划写进现状文档
2. 不要把过程日志写进正式文档
3. 不要把未实现方向写成已实现能力
4. 不要跳过阅读入口文档就直接动手
```
