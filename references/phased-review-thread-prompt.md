# 全项目分阶段代码审查线程模板

## 适用场景

在文档体系已经搭好之后，用户希望另起一个“全项目代码审查线程”，并且要求必须使用子代理协作时，直接交付这份提示词。

## 交付方式

把下面整段提示词原样或按项目实际轻微替换占位符后交给用户。

```text
你现在进入 `<项目名>` 的全项目分阶段代码审查线程。

本线程的目标是：基于当前真实代码和正式文档，对整个仓库做分阶段代码审查，优先识别 bug、风险、行为回归、缺失校验、边界漂移和文档失真。

这次审查有一个硬要求：
必须使用子代理协作。
如果你当前没有子代理能力，必须在开头明确说明“当前环境不具备子代理协作能力，无法按本模板完整执行”，然后停止冒充完成整套审查。

请严格按下面方式工作：

一、审查前校准
1. 先定位项目根目录：`<仓库根目录>`
2. 找到总入口文档：`<总入口文档路径>`；如果名称不同，找等价入口
3. 按入口顺序至少读取：
   - 现状文档
   - 阶段判断文档
   - 未来路径文档
   - 当前版本实施文档
   - 任务链索引
   - 各主要子系统实现事实文档
4. 再读取当前 `git status`、主分支状态和最近活跃改动区域
5. 审查时要始终记住：
   - 现状文档定义“现在真实是什么”
   - 未来路径和任务链不是现状缺陷清单
   - 不要把尚未实现的规划误报成 bug

二、必须分阶段，不要一把扫完
把审查拆成阶段，例如：
1. 阶段一：主入口、主路径、主流程
2. 阶段二：各核心子系统
3. 阶段三：跨模块契约、配置、数据流、权限、安全
4. 阶段四：测试、构建、校验链
5. 阶段五：文档与代码是否失真

你可以按仓库实际重排阶段，但必须先说明本次分阶段方案。

三、必须使用子代理协作
1. 先在主线程做总体切片，不要把阻塞性的核心判断整包丢给子代理
2. 再为不同子系统或不同审查角度分配子代理
3. 每个子代理必须有清晰范围，例如：
   - 后端主路径审查
   - 前端主路径审查
   - 基础设施与配置审查
   - 测试与校验链审查
   - 文档与代码一致性审查
4. 子代理默认只做审查，不做代码修改
5. 子代理回报必须包含：
   - 问题标题
   - 风险说明
   - 触发条件
   - 文件路径与行号
   - 建议优先级

四、主线程必须承担的工作
1. 负责制定分阶段审查方案
2. 负责识别跨模块问题和集成风险
3. 负责对子代理结论去重、归并、提级或降级
4. 负责判断哪些问题是真缺陷，哪些只是未来规划尚未落地
5. 负责输出最终审查报告

五、最终输出格式
最终报告必须“问题优先”，不要先写仓库概述。

请按下面顺序输出：
1. 审查范围与阶段覆盖情况
2. 审查发现
   - 按严重度排序
   - 每条都要包含：标题、影响、触发条件、文件路径、行号、为什么重要
3. 开放问题或判断前提
4. 剩余风险与未覆盖区
5. 如果没有发现问题，要明确写“本轮未发现明确缺陷”，同时说明盲区和剩余风险

六、严格边界
1. 不要把未来路径里的计划当成当前缺陷
2. 不要把过程日志里的临时判断当成正式事实
3. 不要只看文档不看代码
4. 不要只听子代理结论而不做主线程归并
5. 不要把“总结仓库”冒充成“完成审查”
```
